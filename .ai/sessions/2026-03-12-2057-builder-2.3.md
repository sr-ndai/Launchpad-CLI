## Objective

## What Was Done

- Replaced the placeholder submit command with a real CLI flow that resolves
  config, detects or validates the solver, packages the selected input set,
  builds a remote job layout, uploads and extracts the archive, renders a
  generated SLURM script for dry runs, and submits the job in execute mode.
- Added Rich display helpers for the submit dry-run preview and the successful
  submission confirmation so the first operator-facing Phase 2 workflow has a
  consistent terminal presentation.
- Added focused command tests covering dry-run preview behavior, successful
  remote-flow wiring, and the unsupported-input failure path.
- Updated the README and architecture documentation to describe the new submit
  orchestration and dry-run UX.

## Observations

- The most conservative interpretation of this task was to ship the submit
  command only for the already-supported Phase 2 primitives. Status polling,
  logs, result download, and broader machine-readable output remain out of
  scope.
- Explicit `ansys` selection still fails through the adapter's intentional
  `NotImplementedError`, which the command surfaces as a user-facing CLI error
  instead of guessing unsupported runtime behavior.
- Verification is isolated to local tests and command help output. Live SSH,
  upload, archive extraction, and `sbatch` behavior still require real cluster
  access outside this repository.
