## Objective

Fix the remaining Phase 8 scheduler-access regression by routing Launchpad's
shared SLURM commands through the cluster login shell and aligning `launchpad
doctor` with that same scheduler environment.

## Outcome

Outcome: READY_FOR_REVIEW

## What Was Done

- Added a shared SLURM login-shell execution layer in
  `src/launchpad_cli/core/slurm.py` and routed `sbatch`, `squeue`, `sacct`,
  and `scancel` through it.
- Kept the existing scheduler command strings intact and only wrapped them with
  the shared login-shell execution path.
- Added scheduler-specific command-not-found guidance so missing SLURM binaries
  now point operators toward head-node login-shell initialization and
  `remote_binaries.*` absolute paths where those overrides exist.
- Reworked `launchpad doctor` so `sbatch`, `squeue`, and `sacct` validation
  uses the same login-shell scheduler environment while `tar`, `zstd`, and the
  writable-root probe remain on the normal non-interactive exec path.
- Updated regression coverage for the shared wrapper, doctor’s split scheduler
  checks, and the `cancel` path, then refreshed the user docs to explain the
  login-shell requirement.
