## What Was Done

- Added the lightweight `questionary` dependency and wired `launchpad logs` to
  open a TTY-only interactive picker when a human requests a multi-task job
  without `TASK_REF`.
- Styled the picker flow to match the Phase 7 shell language with a pre-picker
  summary panel plus aligned task rows showing label, alias, raw task ID,
  state, selected log kind, and resolved filename.
- Updated follow mode to use retry-by-name `tail` semantics so a selected log
  can be followed even when the file appears shortly after the command starts.
- Refreshed logs help/examples and the targeted docs pages to explain task
  refs, `--log-kind`, legacy fallback limits, and the new picker behavior.
- Expanded regression coverage for picker gating, picker row metadata, final
  logs help text, and follow-mode command construction.

## Observations

- I used the most conservative interactive gate: the picker only opens when
  both stdin and stdout are TTYs and the command is not in `--json` mode, so
  scripts and non-interactive terminals keep deterministic failure behavior.
- Legacy jobs without `launchpad-manifest.json` still participate in the picker
  when the requested log kind can be resolved, but they use placeholder alias
  values and derived labels because no authoritative submitted task metadata
  exists for those historical runs.
