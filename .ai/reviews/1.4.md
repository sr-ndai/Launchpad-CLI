# Review: 1.4

**Verdict:** REVISION_NEEDED

## Summary

Task `1.4` adds real `launchpad ssh` and `launchpad doctor` commands, and the
prompt verification passed (`uv run pytest`, `uv run launchpad ssh --help`,
`uv run launchpad doctor --help`). However, the current diagnostics are not yet
correct enough to accept because the config-validity check reports success even
when the required SSH connection fields are missing.

## Issues

1. `launchpad doctor` marks the config check as `pass` even when both
   `ssh.host` and `ssh.username` are unresolved. In
   `src/launchpad_cli/cli/doctor.py:157`, `_config_resolution_check()` always
   returns `status="pass"` and only interpolates `<missing>` into the message.
   A direct check on this branch confirms the bug:
   `_config_resolution_check((), LaunchpadConfig())` returns `pass` with the
   detail `Config resolved from defaults only; ssh.host=<missing>, ssh.username=<missing>.`
   That contradicts the prompt's requirement for config-validity diagnostics
   and would mislead users into thinking their configuration is valid when the
   command cannot actually establish an SSH session.

2. There is no regression test covering the missing-host / missing-username
   case. `tests/test_operator_commands.py:51` exercises JSON output and
   `tests/test_operator_commands.py:85` covers missing SSH keys, but nothing
   asserts that invalid SSH connection fields cause the config diagnostic to
   fail. Without that coverage, this bug can reappear unnoticed.

## Required Changes

1. Change the config diagnostic so that missing required SSH connection fields
   (`ssh.host`, `ssh.username`, and any other fields you consider mandatory for
   a valid operator-command session) produce a failing diagnostic with a clear
   fix suggestion, instead of a passing result with `<missing>` placeholders.

2. Add regression coverage that proves the config diagnostic fails when the
   required SSH fields are absent, and keep the current passing-path coverage
   for valid configurations.
