# Review: 2.3

**Verdict:** REVISION_NEEDED

## Summary

Task `2.3` wires the first functional `launchpad submit` flow and its dry-run
presentation, and the listed verification commands pass. However, the command
still leaks raw AsyncSSH failures instead of rewriting them into a user-facing
CLI error, which breaks the project’s stated error-handling posture for real
submit execution.

## Issues

1. `launchpad submit` does not catch `asyncssh.Error` from the execute path, so
   real SSH/auth/transport failures will escape as uncaught exceptions instead
   of a `ClickException` message. The current try/except in
   `src/launchpad_cli/cli/submit.py:158` only catches `RuntimeError`,
   `OSError`, `ValueError`, and `NotImplementedError`, but `asyncssh.Error`
   inherits directly from `Exception`. A simple reproduction is monkeypatching
   `_execute_submit()` to raise `asyncssh.Error(1, "boom")`: `CliRunner`
   returns an `Error` exception with no user-facing output.
2. `tests/test_submit.py:49` covers only the successful execute path, and
   `tests/test_submit.py:92` covers only the no-input planning failure. There
   is no regression coverage for the SSH/submission exception path, so the
   uncaught exception behavior is currently unprotected.

## Required Changes

1. Update `launchpad submit` to convert `asyncssh.Error` from the execute path
   into a `click.ClickException`, matching the rest of the command’s human-
   facing error handling.
2. Add a submit-command test which forces `_execute_submit()` to raise an
   AsyncSSH error and asserts the CLI surfaces a normal error message instead of
   leaking the exception object.
