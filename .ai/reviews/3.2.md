# Review: 3.2

**Verdict:** REVISION_NEEDED

## Summary

Task `3.2` adds the first functional `launchpad status` command, the Rich
overview/detail renderables, and the watch loop, but the specific-job flow is
still too brittle. `launchpad status <JOB_ID>` currently aborts if either
`squeue` or `sacct` fails, even when the other source already has enough data
to render the job. Coordinator verification passed for the prompt checks:
`uv run pytest tests/test_status.py`, `uv run launchpad status --help`, and
`uv run pytest`.

## Issues

1. `src/launchpad_cli/cli/status.py:262` hard-requires both `query_squeue()`
   and `query_sacct()` to succeed for specific-job queries. That breaks two
   realistic cases:
   - completed jobs can fail if `squeue --jobs <id>` reports “invalid job id”
     before `sacct` is consulted
   - running jobs can fail on clusters where accounting is unavailable even
     though the live `squeue` row is enough to render the status view
2. `tests/test_status.py:52` covers overview/json/watch wiring, but there is no
   regression coverage for the job-detail fallback behavior when one scheduler
   query fails and the other succeeds.

## Required Changes

1. Make the job-detail collection path degrade gracefully:
   - if `squeue` cannot find the job but `sacct` returns accounting rows, build
     the detail snapshot from accounting data
   - if `sacct` fails but `squeue` returns active rows, render the detail view
     without the accounting-only columns instead of aborting
   - only raise an error when neither source can provide usable job data
2. Add focused tests covering both fallback cases above so the command-level
   behavior stays protected.
