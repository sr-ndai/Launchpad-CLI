# Review: 4.2

**Verdict:** REVISION_NEEDED

## Summary

Task `4.2` replaces the placeholder `launchpad download` command with the
intended Phase 4 orchestration flow, and Coordinator verification passed for
`uv run pytest tests/test_download.py`, `uv run launchpad download --help`,
and `uv run pytest`. The task is not ready to accept yet because the new
command still has one destructive cleanup path that can delete data the user
did not download, and it does not honor the repo's global `--json` contract.

## Issues

1. `src/launchpad_cli/cli/download.py:485` and
   `src/launchpad_cli/cli/download.py:561` delete `plan.remote_job_dir`
   whenever `--cleanup` is set. That is unsafe for combinations the command
   already allows:
   - partial downloads from non-terminal jobs
   - narrowed downloads using `--tasks`
   - narrowed downloads using `--exclude`
   In those cases the command can remove active job data or remote results the
   user intentionally chose not to download.
2. `src/launchpad_cli/cli/download.py:135` defines `DownloadResult.as_dict()`,
   but the command never reads the root CLI's global `--json` option and always
   emits the Rich completion panel. That breaks the documented global CLI
   behavior for a command that clearly has a structured result shape.

## Required Changes

1. Add a safety gate before confirmation so `--cleanup` is rejected unless the
   download is for the full job payload with all selected rows in terminal
   states and no narrowing options are active. Add focused regression coverage
   for the unsafe combinations above.
2. Implement command-level JSON output for `launchpad download` using the
   existing global flag pattern already used by commands such as `status` and
   `doctor`, and cover it with focused CLI tests.
