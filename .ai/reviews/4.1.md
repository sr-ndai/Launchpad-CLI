# Review: 4.1

**Verdict:** REVISION_NEEDED

## Summary

Task `4.1` adds the intended Phase 4 local, archive, and remote filesystem
primitives, and the prompt verification passed for
`uv run pytest tests/test_remote_ops.py tests/test_compress.py tests/test_local_ops.py`
and `uv run pytest`. The task is not ready to accept yet because the new
helpers still have one path-resolution bug, one remote listing error-handling
bug, and one destructive delete-safety gap that later Phase 4 commands would
inherit.

## Issues

1. `src/launchpad_cli/core/local_ops.py:63` resolves relative paths before it
   expands `~`, so `resolve_download_destination(Path("~/Downloads"), ...)`
   produces a literal `.../~/Downloads` path instead of the user's home
   directory. That breaks a normal `launchpad download --output ~/Downloads`
   workflow.
2. `src/launchpad_cli/core/remote_ops.py:203` builds the remote listing command
   as `find ... | sort`. In a normal shell pipeline, the exit status comes from
   `sort`, so a missing or unreadable path can still report success and look
   like an empty listing instead of raising an error.
3. `src/launchpad_cli/core/remote_ops.py:238` shells out to `rm -rf` for any
   normalized path. `PurePosixPath("")` normalizes to `.`, so an empty target
   can become `rm -rf -- .`, and `/` is also currently allowed. That is too
   dangerous for the cleanup groundwork.

## Required Changes

1. Fix `resolve_download_destination()` so user-home paths are expanded before
   the absolute/relative branch selection, and add focused regression coverage
   for a `~/...` destination.
2. Rework `list_remote_directory()` so remote `find` failures are preserved.
   Sorting should not mask a failing exit status. Add a regression test for the
   failure path.
3. Add local guardrails to `delete_remote_path()` so it rejects dangerous
   targets such as `""`, `"."`, and `"/"` before calling `conn.run`, and cover
   those cases with tests.
