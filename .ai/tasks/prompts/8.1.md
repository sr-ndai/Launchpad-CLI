# Task Prompt: 8.1 - Job manifest and task references

## Objective

Establish the submitted metadata contract for Phase 8 by adding a remote job
manifest, stable per-task references, and the first human-facing submit/status
surfaces that expose those references.

## Context

- Phase 7 is merged on `main`, so the next work should build on the current
  command shell and docs baseline rather than the stale Phase 7 branch.
- Batched submits currently rely on raw SLURM array task IDs and a plain
  `input-manifest.txt`; there is no authoritative job metadata artifact for
  later commands to consume.
- `logs` currently infers solver-log extensions from `run_name`, but later
  tasks should instead resolve from submitted metadata captured at submit time.
- Relevant code paths include `src/launchpad_cli/cli/submit.py`,
  `src/launchpad_cli/core/slurm.py`, `src/launchpad_cli/cli/status.py`,
  `src/launchpad_cli/core/config.py`, and `src/launchpad_cli/solvers/`.

## In Scope

- Add per-solver log-catalog configuration under `solvers.<solver>.logs`, with
  Nastran defaulting to `solver = ".f06"` and `telemetry = ".f04"`.
- Extend the solver/config plumbing as needed so submit can capture the
  resolved solver-log catalog for the active job.
- Introduce a remote `launchpad-manifest.json` artifact with top-level
  `version`, `solver`, `logs`, and `tasks`.
- Generate stable task metadata during submit: raw `task_id`, zero-padded
  `alias`, `input_relative_path`, `input_filename`, `input_stem`,
  `display_label`, and `result_dir`.
- Make filename stem the default display label and disambiguate duplicate stems
  deterministically from relative path data instead of guessing.
- Update submit dry-run/success and status human output so multi-input jobs
  show label, alias, and numeric task ID together.
- Add or update tests for config normalization, manifest generation, submit
  behavior, and status rendering.

## Out of Scope

- `logs` selector expansion, `--log-kind`, or interactive picker behavior
- `download` or `cancel` task-selector changes
- Encoding solver metadata into the SLURM comment
- Changing existing JSON payload shapes beyond additive manifest-backed fields
  that the prompt explicitly requires

## Deliverables

- A stable `launchpad-manifest.json` contract written for new submissions
- Multi-input submit/status UX that surfaces the new task references
- Tests covering the new manifest and task-reference groundwork
