# Task Prompt: 5.2 - Single-stream hardening and resume improvements

## Objective

Harden the current single-stream transfer path and interrupted-transfer
behavior so submit and download remain correct, recoverable, and clear for
operators while real-cluster transfer validation remains deferred.

## Context

- Task `5.1` now leaves a deferred-validation plan and a provisional decision
  to keep the existing single-stream path until real-cluster evidence exists.
- The current transfer engine lives in `src/launchpad_cli/core/transfer.py`,
  with configuration in `src/launchpad_cli/core/config.py` and integration
  points in `src/launchpad_cli/cli/submit.py` and
  `src/launchpad_cli/cli/download.py`.
- The roadmap still prioritizes correctness, resume support, and integrity over
  raw throughput. Without real-cluster validation, Phase 5 should not ship a
  speculative parallel-transfer implementation.

## In Scope

- Tighten the current single-stream transfer implementation and codify the
  provisional decision to keep it as the active Phase 5 baseline.
- Update configuration surfaces only where single-stream hardening requires
  them, keeping safe defaults for everyday CLI use.
- Harden interrupted upload/download handling, resume semantics, and operator-
  facing error messages around partial transfers.
- Add or update tests and documentation that cover the selected transfer path
  and its recovery guarantees.

## Out of Scope

- CLI typo suggestions or broad `--json` coverage work assigned to later tasks.
- Logging audit beyond the modules directly touched by the transfer changes.
- Solver-specific feature work.
- Implementing unvalidated parallel-transfer behavior before real-cluster
  benchmark access exists.

## Deliverables

- `src/launchpad_cli/core/transfer.py`
- `src/launchpad_cli/core/config.py`
- `src/launchpad_cli/cli/submit.py`
- `src/launchpad_cli/cli/download.py`
- Associated tests and documentation updates
