# Task Prompt: 9.5 - Logs and utility command redesign

## Objective

Move the human-readable `launchpad logs`, `launchpad ls`, `launchpad cancel`,
and `launchpad cleanup` surfaces onto the Phase 9 restrained visual grammar
while preserving existing command behavior, selector semantics, and
machine-readable paths.

## Context

- The active coordination branch is `phase/09-cli-visual-overhaul`.
- Tasks `9.1` through `9.4` are merged. The shared display primitives, updated
  key/value contrast, submit/status redesign, and download/transfer feedback
  redesign are already on the phase branch.
- The remaining operator-facing commands still carry older panel-heavy or
  command-local presentation patterns. Phase 9 now needs to unify logs and the
  utility flows around the same restrained grammar before help/docs cleanup.
- The detailed design reference remains
  `.ai/plans/LAUNCHPAD_UI_REDESIGN_PLAN.md`, especially sections:
  `3.6 launchpad logs`, `3.7 launchpad cancel`, `3.8 launchpad ls`, and
  `3.9 launchpad cleanup`.
- The logs redesign should be especially conservative: once raw log content
  starts streaming or following, the chrome should mostly disappear so the user
  is reading the log, not the CLI wrapper.

## In Scope

- Redesign human-readable `launchpad logs` header/output so snapshot and follow
  flows use a minimal header and keep raw log reading unobstructed.
- Redesign `launchpad cancel` confirmation and result output around restrained
  confirmation copy, inline status changes, and clearer destructive-flow
  feedback.
- Redesign `launchpad ls` populated and empty states around the new borderless
  listing grammar.
- Redesign `launchpad cleanup` candidate listing, selection prompt, and result
  output around the restrained Phase 9 utility-command style.
- Keep existing selectors, aliases, JSON output, destructive semantics,
  `--yes`, `--follow`, and other command behavior unchanged.
- Update or add regression coverage for the redesigned human-readable logs and
  utility-command surfaces.

## Out of Scope

- Redesigning `download`, `submit`, `status`, `config`, or `doctor`
- Changing log resolution behavior, cleanup selection semantics, remote file
  traversal, cancellation logic, or selector interpretation beyond display-only
  adjustments
- Changing machine-readable outputs or existing confirmation defaults
- Docs/README refresh beyond inline comments or helper notes needed to support
  the implementation

## Deliverables

- Phase 9 human-readable `logs` header/streaming surface
- Phase 9 human-readable `ls`, `cancel`, and `cleanup` output
- Regression coverage for the updated logs and utility-command surfaces

## Revision 2026-03-15

This revision supersedes the original logs-and-utility scope before Builder
implementation begins. Task `9.5` now becomes the shared display hierarchy
revision and retrofit pass for the already-migrated Phase 9 surfaces. The
branch name stays unchanged for continuity, but the execution scope below is
the authoritative task contract.

### Revised Objective

Correct the Phase 9 text hierarchy system across the shared display primitives
and the already-migrated command surfaces so the CLI stops collapsing labels,
values, hints, and suggestions into the same washed-out brightness band.

### Revised Context

- Expert feedback in `.ai/plans/LAUNCHPAD_UI_REVISION.md` identified the main
  remaining UI issue: the flat layout is correct, but the text hierarchy is
  still too dim and uniform outside the welcome screen.
- Tasks `9.2` through `9.4` are already merged, so the fix must retrofit
  `doctor`, `config init`, `config show`, `submit`, `status`, and `download`
  rather than waiting for a final polish pass.
- The structural Phase 9 rules remain unchanged: keep the one-hero-panel max,
  borderless tables, restrained sections, no machine-interface changes, and
  welcome-only branding.
- Treat `.ai/plans/LAUNCHPAD_UI_REVISION.md` as the normative correction for
  hierarchy, tinted neutrals, long-detail handling, and suggestion treatment.

### Revised In Scope

- Update `src/launchpad_cli/display.py` to enforce a five-tier hierarchy:
  primary bold identifiers, default-bright values, secondary tinted-neutral
  labels, tertiary hints/suggestions, and faint rule/separator text.
- Replace remaining semantic uses of generic dim-style detail text in the
  shared display helpers with explicit secondary/tertiary/faint tiers.
- Change shared key/value rendering so labels stay secondary, values return to
  default-bright text, and the default label width increases to `20`.
- Add a multiline-capable shared status-entry helper for doctor-style output so
  long details drop below the label after `max_inline_detail=45` instead of
  turning into wall-of-text status rows.
- Add a shared suggestion/action helper that renders recovery guidance with the
  `→` affordance in the tertiary tier.
- Retrofit the already-migrated Phase 9 command surfaces to the corrected
  hierarchy:
  `doctor`, `config init`, `config show`, `submit`, `status`, and `download`.
- Compress repeated skipped-state messaging in `doctor` to short
  parenthetical reasons rather than repeating full sentences.
- Update or add regression coverage for the hierarchy primitives and the
  retrofitted command outputs without changing JSON or selector expectations.

### Revised Out of Scope

- Redesigning `logs`, `ls`, `cancel`, or `cleanup` in this task; that work now
  moves to task `9.6`
- Changing command behavior, flags, JSON payloads, exit codes, no-color
  semantics, or non-TTY behavior
- Revisiting the welcome screen beyond incidental token-level changes caused by
  shared theme updates
- Docs/README refresh beyond helper comments needed to support the display
  revision

### Revised Deliverables

- Corrected shared display hierarchy primitives
- Retrofitted Phase 9 outputs for `doctor`, `config`, `submit`, `status`, and
  `download`
- Regression coverage for the hierarchy revision and retrofitted surfaces
