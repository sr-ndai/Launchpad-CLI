# Task Prompt: 9.5 - Logs and utility command redesign

## Objective

Move the human-readable `launchpad logs`, `launchpad ls`, `launchpad cancel`,
and `launchpad cleanup` surfaces onto the Phase 9 restrained visual grammar
while preserving existing command behavior, selector semantics, and
machine-readable paths.

## Context

- The active coordination branch is `phase/09-cli-visual-overhaul`.
- Tasks `9.1` through `9.4` are merged. The shared display primitives, updated
  key/value contrast, submit/status redesign, and download/transfer feedback
  redesign are already on the phase branch.
- The remaining operator-facing commands still carry older panel-heavy or
  command-local presentation patterns. Phase 9 now needs to unify logs and the
  utility flows around the same restrained grammar before help/docs cleanup.
- The detailed design reference remains
  `.ai/plans/LAUNCHPAD_UI_REDESIGN_PLAN.md`, especially sections:
  `3.6 launchpad logs`, `3.7 launchpad cancel`, `3.8 launchpad ls`, and
  `3.9 launchpad cleanup`.
- The logs redesign should be especially conservative: once raw log content
  starts streaming or following, the chrome should mostly disappear so the user
  is reading the log, not the CLI wrapper.

## In Scope

- Redesign human-readable `launchpad logs` header/output so snapshot and follow
  flows use a minimal header and keep raw log reading unobstructed.
- Redesign `launchpad cancel` confirmation and result output around restrained
  confirmation copy, inline status changes, and clearer destructive-flow
  feedback.
- Redesign `launchpad ls` populated and empty states around the new borderless
  listing grammar.
- Redesign `launchpad cleanup` candidate listing, selection prompt, and result
  output around the restrained Phase 9 utility-command style.
- Keep existing selectors, aliases, JSON output, destructive semantics,
  `--yes`, `--follow`, and other command behavior unchanged.
- Update or add regression coverage for the redesigned human-readable logs and
  utility-command surfaces.

## Out of Scope

- Redesigning `download`, `submit`, `status`, `config`, or `doctor`
- Changing log resolution behavior, cleanup selection semantics, remote file
  traversal, cancellation logic, or selector interpretation beyond display-only
  adjustments
- Changing machine-readable outputs or existing confirmation defaults
- Docs/README refresh beyond inline comments or helper notes needed to support
  the implementation

## Deliverables

- Phase 9 human-readable `logs` header/streaming surface
- Phase 9 human-readable `ls`, `cancel`, and `cleanup` output
- Regression coverage for the updated logs and utility-command surfaces
