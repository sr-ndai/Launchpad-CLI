# Task Prompt: 5.1 - Transfer benchmark and strategy decision

## Objective

Benchmark Phase 5 transfer candidates against the real cluster workflow and
leave the repository with a documented, evidence-backed recommendation for the
transfer strategy Launchpad should carry forward.

## Context

- `.ai/plan.md` sections 5.1 through 5.4 define the current single-stream
  baseline, the candidate Phase 5 strategies, and the non-negotiable emphasis
  on resume correctness and stability.
- The current runtime uses resumable single-stream SFTP in
  `src/launchpad_cli/core/transfer.py`, wired through
  `src/launchpad_cli/cli/submit.py` and `src/launchpad_cli/cli/download.py`.
- Phase 5 should not guess. The task must either recommend a faster strategy
  with measured evidence or explicitly justify keeping the single-stream
  implementation.

## In Scope

- Benchmark at least three candidate strategies from `.ai/plan.md` section 5.2
  against the actual cluster/head-node environment.
- Capture the benchmark setup, commands, raw measurements, resume behavior,
  observed failure modes, and final recommendation in a repo-tracked document.
- Add any non-user-facing helper script or reproducibility notes needed to run
  the benchmark again.
- If the benchmark cannot be completed because cluster access or prerequisites
  are unavailable, document the exact blocker and stop with a Builder session
  note whose outcome is `BLOCKED`.

## Out of Scope

- Changing the default production transfer behavior in the submit or download
  commands.
- CLI-wide polish items reserved for later Phase 5 tasks.
- Solver-specific workflow changes.

## Deliverables

- `docs/transfer-benchmark.md`
- Any helper script or supporting notes added to make the benchmark
  reproducible

### Revised Scope

- Review successful real-world precedents for SSH/SFTP-based fast transfer,
  using official docs or project sources where possible. At minimum, cover the

  - single large payload transfer with multiple lanes or stripes
  - many-file transfer with a worker pool
  - request pipelining within a single SFTP file handle
- Update `docs/transfer-benchmark.md` so it no longer reads as a blocked
  benchmark note. It should become the implementation brief for the chosen

  - `single-file` mode versus `multi-file` mode

  - how submit and download should select or force each mode
  - the safe implementation strategy for striped single-file upload/download

- Spell out the follow-on work for task `5.2`, including the transfer-engine
  responsibilities, CLI surface changes, and the key safety/fallback rules.

- Decide how the implementation should react when the requested stream count
