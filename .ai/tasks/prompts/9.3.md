# Task Prompt: 9.3 - Submit and status redesign

## Objective

Redesign the human-readable `launchpad submit` and `launchpad status` surfaces
around the Phase 9 visual grammar while preserving their existing behavior,
live-refresh mechanics, task-reference semantics, and machine-readable output.

## Context

- The active coordination branch is `phase/09-cli-visual-overhaul`.
- Tasks `9.1` and `9.2` are merged. The shared display primitives, restrained
  welcome/help shell, grouped human `config show`, and flattened `doctor`
  output are already on the phase branch.
- `submit` and `status` are the next primary runtime surfaces in the Phase 9
  plan. They still rely on the older Phase 7 layout, including stacked panels,
  table titles, and summary chrome that the redesign is intended to replace.
- The detailed design reference remains
  `.ai/plans/LAUNCHPAD_UI_REDESIGN_PLAN.md`, and the canonical sequencing plan
  is `.ai/plans/09-cli-visual-overhaul.md`.
- `submit` and `status` already centralize much of their human rendering in
  `src/launchpad_cli/display.py`, so this task should build on that structure
  rather than moving presentation back into the command modules.
- Design feedback from the accepted `9.2` surfaces: key/value rows should not
  treat keys like low-priority footnotes. Avoid faint grey keys against bright
  white values; use stronger contrast and emphasis so labels still read as
  primary navigational text.

## In Scope

- Redesign `launchpad submit --dry-run` and the successful submit confirmation
  around the Phase 9 hero-card, borderless-table, inline-metadata, and
  lightweight next-step patterns.
- Redesign `launchpad status` overview, job detail, and `--watch` rendering to
  use the restrained Phase 9 layout while preserving the existing `Live`
  refresh flow.
- Keep manifest-backed task references, aliases, array/task semantics, status
  query behavior, and JSON payloads unchanged.
- Use the shared display primitives in `src/launchpad_cli/display.py` instead
  of introducing new command-local render wrappers.
- Update or add regression coverage for the redesigned human-readable submit
  and status paths.

## Out of Scope

- Redesigning `download`, `logs`, `ls`, `cancel`, or `cleanup`
- Changing submit planning, remote transfer behavior, or status/query logic
  beyond presentation-layer adjustments needed to expose existing data
- Changing `--json`, `--quiet`, `--no-color`, or the existing watch semantics
- Docs/README refresh beyond inline comments or helper notes needed to support
  the implementation

## Deliverables

- Phase 9 human-readable `submit --dry-run` and submit success output
- Phase 9 human-readable `status` overview/detail/watch output
- Regression coverage for the updated submit and status surfaces
