# Task Prompt: 8.8 - Harden SLURM numeric field parsing

## Objective

Fix the field-reported scheduler parsing regression where `launchpad status`
crashes on cluster-specific SLURM JSON because Launchpad tries to coerce a
hostname-like field into an integer.

## Context

- The active coordination branch is
  `phase/08-task-references-and-solver-aware-logs`, and PR `#11` is open.
- Field debugging showed `launchpad status` failing with:
  `invalid literal for int() with base 10: 'simulation-r61-8x-dy-r6i-8xlarge7-2'`
- The shared parser in `src/launchpad_cli/core/slurm.py` currently maps
  `node_list` from `"node_list", "nodes", "nodes_alloc"` and also maps
  `nodes` from `"nodes", "nodes_alloc"` through `_optional_int(...)`.
- That means a host-style `nodes_alloc` string can be reused as both a display
  fallback and an integer source, which is unsafe on clusters whose JSON shape
  differs from the narrow test sample.
- The provided cluster JSON also shows other Slurm 23.11 typed-wrapper fields
  such as `cpus`, `cpus_per_task`, `node_count`, `array_job_id`, and
  `array_task_id` encoded as objects with `{"set": ..., "number": ...}`
  rather than plain scalars, so the parser should treat those shapes as
  first-class inputs instead of dropping them.
- The same shared parser feeds `status`, `logs`, `download`, `cancel`, and
  cleanup-related scheduler lookups, so the fix must live in the parser layer,
  not only in `status`.

## In Scope

- Harden shared SLURM JSON parsing so non-numeric string values in numeric
  fields no longer raise `ValueError`.
- Revisit the `JobStatus` and `JobAccounting` field-path mapping so host-style
  fields such as `nodes_alloc` are not reused as integer sources when that is
  unsafe.
- Support Slurm typed-wrapper objects for scalar numeric and string-like fields
  where the parser currently expects plain ints or strings.
- Preserve existing useful string fallbacks for display fields like node names
  where appropriate.
- Add regression tests that cover the reported host-style value plus the
  typed-wrapper fields from the provided `squeue --json` output, and prove the
  shared parser returns a structured row instead of crashing.
- Update any targeted docs or inline comments only if needed to clarify the
  shared parser expectation for cluster-specific SLURM JSON variants.

## Out of Scope

- Changes to scheduler command execution or login-shell behavior from `8.5`
- Broader `status` UX changes unrelated to the parser crash
- New config keys or CLI flags
- Changes outside the shared SLURM parsing path unless they are required by the
  parser fix

## Deliverables

- Hardened shared SLURM parser that skips non-numeric host-style values in
  integer fields
- Regression coverage for the reported cluster-specific scheduler payload shape
  including typed-wrapper scalar fields
- Any minimal doc/comment updates needed to explain the parser behavior
