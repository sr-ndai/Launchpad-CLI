# Task Prompt: 8.2 - Solver-aware log resolution and selector support

## Objective

Make task-oriented commands consume the submitted Phase 8 metadata by resolving
solver logs and task selectors from the remote manifest instead of from naming
heuristics alone.

## Context

- Task `8.1` establishes `launchpad-manifest.json` as the new metadata source
  of truth for submitted jobs.
- The current `logs` command derives solver-log paths from `run_name` and only
  accepts raw numeric task IDs, which is too weak for filename-based batches.
- The user explicitly wants solver-aware log lookup, configurable non-primary
  solver logs such as Nastran `.f04`, and cross-command task references.
- Relevant code paths include `src/launchpad_cli/cli/logs.py`,
  `src/launchpad_cli/cli/download.py`, `src/launchpad_cli/cli/cancel.py`, and
  any shared metadata/selector helpers introduced in `8.1`.

## In Scope

- Add shared task-selector resolution that accepts raw task ID, alias, input
  stem, exact filename, and relative path.
- Apply that selector resolution to `launchpad logs`, `launchpad download
  --tasks`, and `launchpad cancel JOB_ID [TASK_REFS]...`.
- Add `launchpad logs --log-kind KIND`.
- Keep `--solver-log` as the alias for the primary configured solver log kind
  named `solver`.
- Keep `--solver-log`, `--log-kind`, and `--err` mutually exclusive.
- Resolve solver-log paths from submitted manifest data first, using manifest
  `solver` plus manifest `logs`, not `run_name`.
- For legacy jobs without `launchpad-manifest.json`, keep best-effort fallback
  only for the primary solver log and fail clearly for non-primary log kinds.
- Add or update tests for selector resolution, ambiguity handling, legacy
  fallback behavior, and solver-log path resolution.

## Out of Scope

- Interactive arrow-key task selection in `logs`
- Large docs/help refreshes beyond targeted text needed to keep touched command
  behavior coherent
- Any change to the one-job-per-submit model

## Deliverables

- Manifest-backed selector resolution across `logs`, `download`, and `cancel`
- Solver-aware `logs` behavior with `--log-kind`
- Tests that lock the selector and legacy-fallback rules in place
