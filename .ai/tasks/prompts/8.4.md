# Task Prompt: 8.4 - Cluster access diagnostics and Windows SSH

## Objective

Fix the two field-reported cluster-access regressions before Phase 8 merges:
make `launchpad doctor` validate the same remote exec environment used by
Launchpad commands, and make `launchpad ssh` work on Windows by using the local
OpenSSH client instead of AsyncSSH stdio redirection.

## Context

- The active coordination branch is
  `phase/08-task-references-and-solver-aware-logs`, and PR `#9` is already
  open; this task is a follow-up required before that PR should merge.
- Field debugging showed that `doctor` can report `Remote binaries are missing`
  even when an interactive manual SSH login can resolve them. The current
  implementation forces `sh -lc` for doctor probes, while the rest of
  Launchpad uses the SSH exec shell via `conn.run(...)`.
- Field debugging on Windows showed `launchpad ssh` failing with
  `SSH session failed: [WinError 6] the handle is invalid` and a traceback
  mentioning `_proactorReadPipeTransport` and `empty_waiter`. The current code
  passes `sys.stdin`, `sys.stdout`, and `sys.stderr` into AsyncSSH process
  redirection, which is not reliable on the Windows Proactor event loop.
- Manual native `ssh` already works on the affected Windows workstation, so the
  supported fix is to use the local OpenSSH client for `launchpad ssh` on
  Windows only.

## In Scope

- Update doctor's remote binary and writable-root probes so they execute
  through the same remote exec environment Launchpad uses for submit/status/
  download flows rather than forcing `/bin/sh -lc`.
- Improve doctor's remote-binary failure text so it explains that Launchpad's
  remote exec environment could not resolve the binary and points users toward
  `remote_binaries.*` absolute paths or non-interactive shell PATH fixes.
- Replace the Windows-only `launchpad ssh` interactive path with a local
  `ssh` subprocess which maps existing config values to command-line arguments.
- Honor existing SSH config for the Windows subprocess path:
  `host`, `port`, `username`, `key_path`, and optional `known_hosts_path`.
- Fail clearly if the local `ssh` executable is unavailable on Windows.
- Add or update regression tests for the doctor probe alignment and the Windows
  SSH behavior.
- Update the relevant command/config/troubleshooting docs to describe the new
  Windows `launchpad ssh` behavior and the non-interactive PATH expectations
  behind doctor's remote-binary checks.

## Out of Scope

- Adding new SSH config fields, CLI flags, or environment variable aliases
- Broad remote-command refactors outside the doctor probes required here
- Replacing AsyncSSH for non-interactive commands or for non-Windows
  `launchpad ssh`
- Changing how interactive manual SSH logins initialize the user's shell on the
  cluster

## Deliverables

- Doctor probes aligned with Launchpad's normal remote exec behavior
- Clearer doctor failure messaging for missing remote binaries
- A Windows `launchpad ssh` implementation which shells out to local OpenSSH
- Regression coverage for both bug fixes
- Docs updates covering the new behavior and operator guidance
