# Task Prompt: 3.1 - SLURM status parsing and query primitives

## Objective

Implement the reusable SLURM query and parsing layer required for Phase 3
monitoring commands.

## Context

- Phase 3 starts with structured SLURM data handling because `launchpad status`,
  `launchpad logs`, and `launchpad cancel` all depend on reliable scheduler
  metadata.
- `.ai/plan.md` sections 3.2, 4.2, 7.2, 7.3, 12, and 15 describe the intended
  status behavior, job-tracking model, and testing expectations.
- The current repository already contains submit helpers in `core/slurm.py`
  from Phase 2, but status/accounting parsing remains unimplemented.

## In Scope

- Extend `core/slurm.py` with the Phase 3 status/accounting data models and
  `squeue --json` / `sacct --json` parsing helpers.
- Add the reusable remote query wrappers needed to fetch scheduler data over the
  established SSH transport layer.
- Add focused tests for parsing behavior and query-helper command construction
  without requiring a live cluster.
- Update documentation where the SLURM query layer is described.

## Out of Scope

- The `launchpad status`, `launchpad logs`, and `launchpad cancel` command
  wiring.
- Live Rich rendering, watch loops, or remote log-tail behavior.

## Deliverables

- `src/launchpad_cli/core/slurm.py`
- Associated tests and documentation updates
