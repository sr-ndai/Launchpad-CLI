# Task Prompt: 8.5 - SLURM login-shell execution and diagnostics

## Objective

Fix the remaining field-reported scheduler-access regression by making
Launchpad execute SLURM commands through the login-shell environment available
on the cluster head node, and make `launchpad doctor` validate that same
scheduler environment.

## Context

- The active coordination branch is
  `phase/08-task-references-and-solver-aware-logs`, and PR `#9` remains open.
- Task `8.4` fixed the doctor non-interactive probe mismatch and the Windows
  `launchpad ssh` crash, but new field testing shows `status` and `logs` still
  fail with `squeue: command not found` and `sacct: command not found`.
- The operator reports that these scheduler commands work during manual SSH use
  on the head node, which strongly indicates that the cluster exposes SLURM
  through login-shell initialization rather than the raw non-login exec PATH.
- Launchpad's shared SLURM path currently invokes `sbatch`, `squeue`, and
  `sacct` directly in `src/launchpad_cli/core/slurm.py`, and `cancel` invokes
  `scancel` directly in `src/launchpad_cli/cli/cancel.py`.
- Fixing only `status` and `logs` would leave the same regression in submit,
  cancel, download, and cleanup because they share those helpers.

## In Scope

- Introduce a shared helper for executing SLURM commands through the cluster's
  login-shell environment.
- Use that shared scheduler-exec path for `sbatch`, `squeue`, `sacct`, and
  `scancel`.
- Keep the command strings themselves unchanged apart from the login-shell
  wrapper; preserve the existing JSON parsing and current command flags.
- Re-align `launchpad doctor` so scheduler-binary checks (`sbatch`, `squeue`,
  `sacct`) validate the same login-shell scheduler environment that Launchpad
  now uses.
- Leave non-scheduler doctor checks (`tar`, `zstd`, writable root) on their
  current non-interactive exec path.
- Improve scheduler failure messaging so command-not-found errors point
  operators toward login-shell initialization or explicit absolute paths where
  relevant.
- Add or update regression tests for the shared SLURM execution path, the
  doctor scheduler checks, and affected user-facing command behavior.
- Update the relevant docs to explain that Launchpad runs scheduler commands
  through the login shell because some clusters expose SLURM only there.

## Out of Scope

- New config fields, CLI flags, or environment variable aliases
- Broad changes to non-SLURM remote commands such as `tar`, `zstd`, or general
  filesystem checks
- Changes to the Windows `launchpad ssh` implementation landed in `8.4`
- Changing scheduler output parsing or command semantics beyond the shell
  wrapper and clearer operator messaging

## Deliverables

- Shared login-shell execution for scheduler commands
- Doctor scheduler checks aligned with that login-shell path
- Regression coverage for the shared scheduler-exec behavior
- Docs updates covering the login-shell requirement and recovery guidance
