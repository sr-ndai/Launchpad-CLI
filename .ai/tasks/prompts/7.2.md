# Task Prompt: 7.2 - Guided setup and diagnostics UX

## Objective

Build on the Phase 7 design system so Launchpad's first-run and troubleshooting
surfaces feel guided, actionable, and easy to recover from for a Windows-first
operator.

## Context

- Task `7.1` establishes the shared Launchpad terminal design system and help
  shell that this task should reuse rather than bypass.
- `src/launchpad_cli/cli/config_cmd.py` currently uses plain `click.prompt`
  and `click.echo` flows for `config init`, with no visual summary or next-step
  guidance.
- `src/launchpad_cli/cli/doctor.py` already collects useful diagnostics, but
  the rendered output is still a simple pass/fail list and does not yet feel
  like a guided recovery tool.
- This task is where "easy to use" becomes real for new operators: clear
  prompts, clear results, and clear next actions.

## In Scope

- Redesign `launchpad config init` around the shared Phase 7 display system,
  including clearer prompt copy, a final summary, and explicit next-step
  guidance after writing the user config.
- Redesign `launchpad doctor` output so checks are grouped and the failure or
  skip paths point the operator toward the next useful command or fix.
- Improve the human-facing error or validation messaging touched by these flows
  so missing config or SSH prerequisites read as actionable Launchpad errors
  instead of bare exceptions.
- Add or update tests and docs for the revised setup and diagnostics UX.

## Out of Scope

- Implementing `launchpad config edit` or `launchpad config validate`
- Real-cluster validation beyond the existing fake or local test coverage
- Polishing the submit, status, download, logs, ls, cancel, or cleanup command
  surfaces outside the shared helpers these flows reuse

## Deliverables

- Updated setup and diagnostics command flows
- Tests covering the new guided output and prompt behavior
- Documentation updates for the revised `config init` and `doctor` experience
