# Task Prompt: 1.4 - Operator commands and diagnostics

## Objective

Expose the Phase 1 user-facing operator commands that depend on the config and
transport primitives: `launchpad ssh` and `launchpad doctor`.

## Context

- Phase 1 calls for `launchpad ssh` and `launchpad doctor` after the
  configuration and SSH layers are in place.
- `.ai/plan.md` sections 3.8, 3.10, 4.2, 8, 12, and 15 describe the intended
  behavior and verification posture.
- This task depends on `1.2` and `1.3` being complete and merged.

## In Scope

- Implement `launchpad ssh` as the CLI entry point for interactive SSH
  passthrough using the established config and logging infrastructure.
- Implement `launchpad doctor` diagnostics for config validity, SSH
  reachability checks, remote binary checks, and writable path checks where
  the underlying primitives exist.
- Add tests for command wiring and any logic that can be validated without the
  real cluster.
- Update documentation for these operator commands.

## Out of Scope

- Submission, status, logs, download, cleanup, or remote listing workflows.
- New config schema work beyond what is needed to consume the established
  configuration APIs.

## Deliverables

- `src/launchpad_cli/cli/ssh_cmd.py`
- `src/launchpad_cli/cli/doctor.py`
- Associated tests and documentation updates

## Branch

`task/1.4-operator-commands-diagnostics`

## Verification

- `uv run pytest`
- `uv run launchpad ssh --help`
- `uv run launchpad doctor --help`
